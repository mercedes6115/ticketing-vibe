## 1. 대기열은 왜 SSE를 선택했는가

### 고려한 대안

- 짧은 주기의 HTTP 폴링
- WebSocket 하나로 모든 실시간 이벤트 통합
- SSE

### 선택

- 대기열 상태 전달에는 `SSE`를 사용했습니다.

### 이유

- 대기열은 서버에서 클라이언트로 상태를 단방향 push 하면 충분합니다.
- 좌석 선택처럼 양방향 상호작용이 필요한 영역이 아니므로 WebSocket까지 요구되지 않습니다.
- 구현 복잡도와 연결 관리 비용을 비교했을 때, 순번 갱신과 토큰 발급 알림에는 SSE가 더 단순합니다.

### 감수한 비용

- 브라우저 `EventSource` 제약 때문에 인증 헤더를 직접 붙이기 어렵습니다.
- 그래서 JWT를 query parameter로 전달하는 별도 처리 로직이 필요합니다.
- 네트워크 환경에 따라 연결 재시도나 장시간 유지에 대한 운영 고려가 필요합니다.

## 2. 좌석 상태와 대기열 채널을 왜 분리했는가

### 선택

- 대기열은 `SSE`
- 좌석 상태 변경은 `WebSocket(STOMP)`

### 이유

- 좌석 상태는 특정 이벤트 방에 여러 사용자가 함께 구독하는 브로드캐스트 성격이 강합니다.
- 한 사용자의 선점, 만료, 판매 완료가 다른 사용자 화면에도 즉시 반영돼야 하므로 pub/sub 모델이 자연스럽습니다.
- 반면 대기열은 개인별 상태 갱신이 중심이라 채널 특성이 다릅니다.

### 감수한 비용

- 실시간 통신 방식이 두 가지라 운영 복잡도가 약간 늘어납니다.
- 프론트엔드도 연결 관리 코드를 두 벌 유지해야 합니다.

## 3. 좌석 선점은 왜 동기 처리로 뒀는가

### 고려한 대안

- 좌석 선점도 Kafka로 비동기 처리
- API 요청 시 즉시 DB와 Redis를 갱신하는 동기 처리

### 선택

- 좌석 홀드는 동기 처리로 뒀습니다.

### 이유

- 사용자는 좌석을 클릭한 직후 성공/실패를 즉시 알아야 합니다.
- 좌석 선점은 인터랙션 지연에 매우 민감하므로, 비동기 큐에 넣으면 UX가 나빠질 가능성이 큽니다.
- 좌석 경쟁 자체는 짧은 크리티컬 섹션으로 다뤄야 하므로 Redisson 분산락으로 직렬화하는 쪽이 더 직접적입니다.

### 감수한 비용

- 피크 구간에는 락 경합이 생길 수 있습니다.
- API 서버가 좌석 선점 경쟁을 직접 받아야 하므로, 매우 큰 규모에서는 병목이 될 수 있습니다.

## 4. 예매 확정은 왜 Kafka 비동기로 분리했는가

### 고려한 대안

- `/api/bookings`에서 DB 쓰기와 결제 처리를 모두 동기 수행
- API는 요청을 수락만 하고 실제 확정은 Consumer가 처리

### 선택

- 예매 확정은 Kafka 기반 비동기 처리로 분리했습니다.

### 이유

- 예매 요청이 몰릴 때 가장 비싼 작업은 DB 쓰기와 후속 처리입니다.
- 이를 API 응답 경로에서 분리하면 `202 Accepted`를 빠르게 반환할 수 있습니다.
- 사용자 체감 응답과 실제 처리 완료를 분리함으로써 피크 상황에서 API 경로를 짧게 유지할 수 있습니다.

### 감수한 비용

- 최종 일관성 모델이 되므로 클라이언트 polling이 필요합니다.
- 즉시 성공/실패가 아니라 `PROCESSING -> CONFIRMED/FAILED` 전이를 관리해야 합니다.
- 장애 분석 시 HTTP만 보는 것이 아니라 Kafka lag와 Consumer 처리도 함께 봐야 합니다.

## 5. 왜 Redis TTL과 DB 상태를 함께 사용하는가

### 고려한 대안

- 좌석 상태를 DB만으로 관리
- Redis만 진실 원천으로 사용
- Redis와 DB를 역할 분리해 함께 사용

### 선택

- Redis와 DB를 함께 사용했습니다.

### 이유

- Redis는 짧은 TTL 기반 홀드와 빠른 상태 조회에 유리합니다.
- DB는 최종 판매 상태와 영속 데이터를 안정적으로 보관하는 데 적합합니다.
- 홀드와 판매를 하나의 저장소에만 몰아넣으면 각각 다른 문제가 생깁니다.
  - DB만 쓰면 짧은 홀드 경쟁에 대한 부담이 큽니다.
  - Redis만 쓰면 영속성과 조회 요구사항이 약해집니다.

### 감수한 비용

- 두 저장소 간 상태 전이를 맞춰야 합니다.
- stale state, 중복 해제, 만료 직전 예매 같은 경계 조건을 직접 설계해야 합니다.

## 6. 왜 좌석 홀드 만료를 TTL 이벤트로 복구하는가

### 고려한 대안

- 주기적인 스케줄러가 만료 좌석을 스캔
- 사용자가 다시 진입할 때만 만료 여부를 확인
- Redis key expiration 이벤트로 즉시 반응

### 선택

- Redis key expiration 이벤트를 사용했습니다.

### 이유

- 스케줄러 스캔보다 더 빠르게 만료를 반영할 수 있습니다.
- 사용자 행동에 의존하지 않고 서버가 자동으로 좌석을 복구할 수 있습니다.
- 실시간 좌석 브로드캐스트와 연결하기 좋습니다.

### 감수한 비용

- Redis 설정과 이벤트 리스너에 대한 운영 의존성이 생깁니다.
- 만료 이벤트 처리 실패 시 복구 경로를 추가로 고민해야 합니다.

## 7. 왜 booking 상태를 Redis에 별도로 저장하는가

### 선택

- `booking:status:{bookingNo}` 키로 `PROCESSING / CONFIRMED / FAILED` 상태를 별도로 유지했습니다.

### 이유

- 비동기 처리 구조에서는 클라이언트가 "요청은 접수됐지만 아직 완료되지 않은 상태"를 조회할 수 있어야 합니다.
- DB에 즉시 row가 없더라도, 처리 중이라는 사실을 Redis에서 빠르게 보여줄 수 있습니다.
- polling API를 단순하게 만들 수 있습니다.

### 감수한 비용

- TTL 만료 후 DB fallback 로직이 필요합니다.
- Redis와 DB 사이 일시적인 상태 차이를 고려해야 합니다.

## 8. 왜 load test를 두 종류로 나눴는가

### 선택

- `load-test-full-flow.js`: 대표 사용자 시나리오
- `load-test-booking.js`: 예매 확정 파이프라인 부분 테스트

### 이유

- 전체 플로우 테스트는 서비스 관점에서 자연스럽지만, 병목 원인을 분리하기 어렵습니다.
- 반대로 부분 테스트는 실제 사용자 흐름과는 거리가 있지만, Kafka Consumer와 예매 확정 경로만 따로 측정할 수 있습니다.
- 두 테스트를 함께 써야 "사용자 경험"과 "내부 처리량"을 분리해서 해석할 수 있습니다.

### 감수한 비용

- 부분 테스트는 인위적으로 준비된 상태를 전제로 하므로, 설명 없이 제시하면 과장처럼 보일 수 있습니다.
- 문서에서 각 시나리오의 목적과 한계를 분명히 적어야 합니다.

## 9. 현재 구조의 한계

현재 구조는 개인 프로젝트와 소규모 서비스 검증에는 적합하지만, 더 큰 규모를 가정하면 다음 고민이 남습니다.

- polling 기반 booking 상태 확인은 요청 수를 늘릴 수 있습니다.
- Redis Sentinel, Kafka, WebSocket, SSE를 함께 운영하므로 복잡도가 낮지 않습니다.
- 좌석 조회가 이벤트 단위 전체 조회라면 좌석 수가 커질수록 API 부담이 커질 수 있습니다.
- 결제는 현재 mock 수준이므로 실제 외부 PG 연동 시 지연과 실패 모델이 더 복잡해집니다.

## 10. 한 줄 정리

이 프로젝트의 핵심 선택은 "좌석 경쟁은 즉시 제어하고, 예매 확정은 비동기 분리하며, 운영 중에는 이를 관찰 가능하게 만든다"입니다.
