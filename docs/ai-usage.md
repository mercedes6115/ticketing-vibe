## 1. 기본 원칙

이 프로젝트에서 AI는 구현 속도를 높이는 보조 도구로 사용했습니다. 하지만 아래 항목은 직접 판단하고 검증하는 것을 원칙으로 삼았습니다.

- 문제 정의
- 아키텍처 방향
- 동시성 제어 방식 선택
- 트레이드오프 판단
- 부하 테스트 시나리오 정의
- 결과 해석과 개선 판단

즉, AI가 초안을 제안할 수는 있어도 최종 설계 책임은 직접 집니다.

## 2. AI를 활용한 영역

### 코드 초안 생성

- 반복적인 DTO, API 호출 코드, 테스트 스크립트 초안
- 문서 구조 초안
- 설명 문장 정리

### 아이디어 탐색

- 대기열, 좌석 홀드, 비동기 예매 구조를 설명하는 문장 정리
- 성능 보고서 템플릿과 체크리스트 초안
- README, 트레이드오프 문서의 뼈대 구성

### 검토 보조

- 기존 테스트 스크립트와 실제 코드가 맞는지 비교하는 체크 포인트 정리
- 포트폴리오 관점에서 빠진 문서 자산 식별

## 3. 직접 수행한 판단

### 문제 정의

이 프로젝트의 중심 문제를 단순 예매 CRUD가 아니라 다음으로 잡았습니다.

- 예매 오픈 순간의 동시 접속
- 동일 좌석 경쟁
- 예매 확정 처리의 병목 분리
- 홀드 만료 후 상태 복구
- 관측 가능한 운영 지표 수집

### 설계 판단

직접 설명 가능해야 하는 핵심 판단은 다음입니다.

- 대기열은 SSE, 좌석 상태는 WebSocket으로 분리
- 좌석 홀드는 동기 처리, 예매 확정은 Kafka 비동기 처리
- Redis TTL과 DB 상태를 함께 사용
- 전체 플로우 테스트와 부분 테스트를 분리

### 검증 판단

- 어떤 부하 테스트가 실제 사용자 시나리오에 가까운지 구분
- `full_flow`는 대표 사용자 시나리오, `booking`은 부분 테스트로 해석
- 메트릭을 단순 수집이 아니라 병목 해석과 연결

## 4. AI 제안과 실제 적용 사이의 검증 방식

AI가 제안한 내용은 그대로 수용하지 않고 아래 기준으로 검증했습니다.

### 1. 현재 코드와 맞는가

- 실제 컨트롤러 요청 형식과 테스트 스크립트가 일치하는지 확인
- enum 값, 인증 방식, Redis 키 사용 방식이 실제 구현과 맞는지 대조

### 2. 실제 시나리오로 성립하는가

- 코드상 호출 가능하더라도 사용자 흐름으로 자연스러운지 분리해 판단
- 부분 테스트를 E2E처럼 과장하지 않도록 설명을 수정

### 3. 측정 가능한가

- 설계 설명이 아니라 실제 메트릭, threshold, 테스트 자산으로 연결되는지 확인

## 5. 이 프로젝트에서 AI를 과신하지 않기 위해 한 일

- API 문서보다 실제 코드 구현을 우선 기준으로 삼음
- 부하 테스트 스크립트가 실서비스 흐름과 얼마나 가까운지 따로 분석함
- README와 성능 문서에서 테스트의 목적과 한계를 명시함
- 포트폴리오 문서에 "무엇을 검증했고 무엇은 아직 남았는지"를 분리해 기록함

## 6. 아직 사람이 직접 더 해야 하는 부분

AI가 도와줄 수 있어도 아래는 직접 결과를 만들어야 합니다.

- 실제 부하 테스트 실행
- Grafana 스크린샷 확보
- 개선 전후 성능 수치 비교
- 배포 환경에서의 검증
- 짧은 데모 영상 또는 설명 자료 정리

## 7. 한 줄 정리

이 프로젝트에서 AI는 구현과 문서화 속도를 높이는 도구였고, 문제 정의와 설계 의도, 성능 검증, 결과 해석은 직접 주도하는 영역으로 두었습니다.
