# 🎟️ 실시간 티켓팅 시스템 기획문서 v1.0
> Tech Stack: React · Spring Boot · MySQL · Redis

---

## 1. 프로젝트 개요

콘서트·스포츠·공연 티켓을 실시간으로 예매할 수 있는 시스템입니다.  
오픈과 동시에 수천 명이 몰리는 트래픽 폭증 상황에서의 **동시성 제어, 대기열 관리, 결제 정합성**을 핵심 기술 과제로 삼습니다.

### 1.1 핵심 목표

- 동시 좌석 선점 → Redis 분산락으로 Race Condition 방지
- 트래픽 폭증 대응 → Redis 대기열(Sorted Set) + SSE 순번 알림
- 결제 정합성 → Saga 패턴(보상 트랜잭션) 구현
- 실시간 좌석 현황 → WebSocket(STOMP) + Redis Pub/Sub

### 1.2 기술 스택

| 구분 | 기술 | 선택 이유 |
|------|------|-----------|
| Frontend | React, TypeScript, Zustand, TailwindCSS | SPA 구조로 실시간 좌석 UI 업데이트에 적합 |
| Backend | Spring Boot 3, Java 17 | 안정적인 트랜잭션 처리 및 생태계 |
| DB | MySQL 8, Redis | MySQL: 정합성 / Redis: 분산락·캐싱·대기열 |
| 실시간 | WebSocket (STOMP), SSE | 좌석 실시간 반영 / 대기열 순번 알림 |
| Infra | Docker, GitHub Actions | 로컬~배포 일관성, CI/CD 자동화 |

## 2. 화면 설계 (Frontend)

React + TypeScript 기반 SPA로 구성하며, 실시간 상태 변화가 핵심인 **좌석 선택 화면**과 **대기열 화면**을 중점 개발합니다.

### 2.1 화면 목록

| 화면명 | 주요 기능 | 구현 난이도 |
|--------|-----------|-------------|
| 로그인 / 회원가입 | 일반 로그인, JWT 발급 | 낮음 |
| 메인 (이벤트 목록) | 이벤트 카드 리스트, 검색·필터 | 낮음 |
| 이벤트 상세 | 공연 정보, 잔여석 현황, 예매하기 버튼 | 중간 |
| **좌석 선택** | 실시간 좌석 맵, WebSocket 연동 | **높음 ★** |
| **대기열** | 현재 순번·예상 대기시간 SSE 수신 | **높음 ★** |
| 결제 | 결제 정보 입력, 결제 처리 (Mock) | 중간 |
| 예매 완료 | 예매 번호·QR 표시 | 낮음 |
| 마이페이지 | 예매 내역 조회, 취소 신청 | 중간 |
| 관리자 - 이벤트 관리 | 이벤트 등록·수정·삭제 | 낮음 |
| 관리자 - 예매 현황 | 예매 통계, 실시간 모니터링 | 중간 |

### 2.2 핵심 화면 상세

#### ① 좌석 선택 화면

- 좌석 구역별 색상 구분 (사용 가능 / 임시선점 / 판매완료)
- WebSocket(STOMP) 구독 → 타인의 좌석 선택 즉시 반영
- 좌석 클릭 시 5분 타이머 시작 → 미결제 시 자동 해제
- 선택 좌석 요약 패널 (구역·가격·잔여 수량)

#### ② 대기열 화면

- 예매 오픈 시 자동으로 대기열 진입 처리
- SSE(Server-Sent Events)로 현재 순번·예상 대기시간 실시간 수신
- Progress Bar 및 순번 애니메이션으로 UX 개선
- 입장 가능 시 `POST /api/queue/token` 호출 → UUID 입장 토큰 발급 (TTL 10분) → 좌석 선택 화면으로 전달
- 입장 가능 시 자동으로 좌석 선택 화면으로 이동

#### ③ 결제 화면

- 선점된 좌석 정보 + 잔여 선점 시간 표시
- 결제 수단 선택 (Mock 처리 - 카드/계좌이체)
- 결제 완료 후 예매 번호 및 QR 코드 발급

### 2.3 상태 관리 전략

- **전역 상태 (Zustand)**: 로그인 유저 정보, 선택 좌석, 대기열 상태
- **서버 상태 (React Query)**: 이벤트 목록, 예매 내역 (자동 캐싱·리페치)
- **실시간 상태**: WebSocket 메시지 수신 시 Zustand store 직접 업데이트

## 3. API 설계 (Backend)

RESTful API 원칙을 준수하며, 실시간 기능은 WebSocket(STOMP)과 SSE를 병행 사용합니다.  
모든 API는 JWT Bearer 토큰 인증이 기본입니다.

### 3.1 인증 API

| Method | Endpoint | 설명 | 주요 처리 |
|--------|----------|------|-----------|
| POST | /api/auth/signup | 회원가입 | 비밀번호 BCrypt 암호화, 이메일 중복 체크 |
| POST | /api/auth/login | 로그인 | JWT Access(15분) + Refresh(7일) 발급 |
| POST | /api/auth/reissue | 토큰 재발급 | Redis에서 Refresh 토큰 검증 후 재발급 |
| POST | /api/auth/logout | 로그아웃 | Redis Refresh 토큰 삭제 |

### 3.2 이벤트 API

| Method | Endpoint | 설명 | 주요 처리 |
|--------|----------|------|-----------|
| GET | /api/events | 이벤트 목록 | 커서 기반 페이지네이션, 카테고리·날짜 필터 |
| GET | /api/events/{id} | 이벤트 상세 | 공연 정보 + 잔여 좌석 수 Redis 캐싱 |
| POST | /api/admin/events | 이벤트 등록 | 이미지 업로드, 좌석 구역 설정 |
| PUT | /api/admin/events/{id} | 이벤트 수정 | 수정 시 캐시 무효화 |

### 3.3 좌석 API ⭐ 핵심

| Method | Endpoint | 설명 | 주요 처리 |
|--------|----------|------|-----------|
| GET | /api/events/{id}/seats | 좌석 목록 조회 | 좌석 상태(available/hold/sold) Redis 조회 |
| POST | /api/seats/{id}/hold | 좌석 임시 선점 | queue:token:{userId} 입장 토큰 검증 → 유효 시 Redis 분산락 획득 → 5분 TTL 임시 선점 |
| DELETE | /api/seats/{id}/hold | 임시 선점 해제 | Lock 해제, 상태 복구 |
| WS | /ws/events/{id}/seats | 실시간 좌석 현황 | STOMP - 좌석 상태 변경 브로드캐스트 |

> ※ WS 행은 WebSocket 엔드포인트 (STOMP 프로토콜)

### 3.4 대기열 API ⭐ 핵심

| Method | Endpoint | 설명 | 주요 처리 |
|--------|----------|------|-----------|
| POST | /api/queue/enter | 대기열 진입 | Redis ZADD로 입장 시각 기준 순번 부여 |
| GET | /api/queue/status | 대기 상태 조회 | 현재 순번, 예상 대기시간 반환 |
| SSE | /api/queue/stream | 대기 순번 스트림 | SSE - 일정 간격으로 순번 Push |
| POST | /api/queue/exit | 대기열 이탈 | Redis ZREM 처리 |
| POST | /api/queue/token | 입장 토큰 발급 | 내 차례 확인 후 UUID 토큰 발급 → Redis queue:token:{userId} TTL 10분 저장 |

> ※ SSE 행은 Server-Sent Events 스트림 엔드포인트

### 3.5 예매 API

| Method | Endpoint | 설명 | 주요 처리 |
|--------|----------|------|-----------|
| POST | /api/bookings | 예매 생성 | 임시선점 확인 → 결제 요청 → 예매 확정 (Saga) |
| GET | /api/bookings/{id} | 예매 상세 조회 | QR 코드 URL 포함 |
| GET | /api/users/me/bookings | 내 예매 목록 | 상태별 필터링, 페이지네이션 |
| POST | /api/bookings/{id}/cancel | 예매 취소 | 취소 정책 검증 → 환불 처리 → 좌석 복구 |

### 3.6 결제 API

| Method | Endpoint | 설명 | 주요 처리 |
|--------|----------|------|-----------|
| POST | /api/payments | 결제 요청 | Mock 결제 처리, 멱등성 키(idempotency key) 적용 |
| POST | /api/payments/{id}/refund | 환불 처리 | 예매 취소 시 연계 호출 |
| GET | /api/payments/{id} | 결제 상세 조회 | 결제 수단, 금액, 상태 반환 |

## 4. 데이터 모델

### 4.1 주요 엔티티

**User**
```
id, email, password, nickname, role(USER/ADMIN), created_at
```

**Event (공연/이벤트)**
```
id, title, description, venue, start_at, open_at, status, total_seats, available_seats
```

**Seat (좌석)**
```
id, event_id, section, row, number, price, status(AVAILABLE/HOLD/SOLD)
```

**Queue (대기열)**
```
→ Redis Sorted Set으로 관리 (DB 테이블 없음)
key: queue:event:{eventId}
score: 진입 timestamp
value: userId
```

**Booking (예매)**
```
id, user_id, event_id, seat_id, status(PENDING/CONFIRMED/CANCELLED), booking_no
```

**Payment (결제)**
```
id, booking_id, amount, method, status(SUCCESS/FAILED/REFUNDED), idempotency_key
```

### 4.2 Redis 키 설계

| Key | 용도 | TTL |
|-----|------|-----|
| `seat:hold:{seatId}` | 선점한 userId | 300초 (5분) |
| `seat:status:{eventId}` | Hash, 좌석 전체 상태 캐싱 | - |
| `queue:event:{eventId}` | Sorted Set, 대기열 | - |
| `queue:token:{userId}` | 입장 토큰 (UUID) ⭐ 신규 | 600초 (10분) |
| `auth:refresh:{userId}` | Refresh 토큰 | 7일 |
| `event:cache:{eventId}` | 이벤트 상세 캐싱 | 60초 |

## 5. 트러블슈팅 포인트

> 포트폴리오의 핵심입니다. 아래 이슈들을 의도적으로 발생시키고 해결 과정을 문서화하세요.

| 이슈 | 원인 | 해결 방법 | 관련 기술 |
|------|------|-----------|-----------|
| 동시 좌석 선점 | Race Condition | Redis 분산락 (Redisson) + TTL 5분 임시선점 | Redis, Redisson |
| 트래픽 폭증 | 오픈 시 동시 요청 급증 | Redis Queue + 토큰 버킷 방식 Rate Limiting | Redis, Spring |
| 결제-예매 정합성 | 결제 성공 후 예매 실패 | Saga 패턴 (보상 트랜잭션) 적용 | Spring Tx |
| N+1 쿼리 | 연관 엔티티 Lazy 로딩 | Fetch Join + QueryDSL DTO Projection | JPA, QueryDSL |
| 실시간 좌석 동기화 | DB와 WebSocket 상태 불일치 | Redis Pub/Sub로 상태 변경 이벤트 발행·구독 | Redis, STOMP |

### 5.1 Redis 분산락 구현 포인트

- Redisson RLock 사용 (직접 Lua Script 구현 대비 안전성↑)
- 락 획득 실패 시 → 409 Conflict 응답 + 클라이언트 재시도 안내
- 락 TTL = 선점 TTL(5분) + 여유 시간(30초)으로 설정

### 5.2 입장 토큰 검증 흐름 ⭐ 신규

```
대기열 순번 도달
  → POST /api/queue/token  (UUID 발급, Redis에 TTL 10분 저장)
  → 프론트엔드가 토큰을 헤더에 담아 좌석 선점 요청
  → POST /api/seats/{id}/hold  (queue:token:{userId} 존재 여부 검증)
  → 토큰 없으면 403 Forbidden  ← 대기열 우회 차단
  → 토큰 있으면 분산락 획득 후 선점 처리
```

### 5.3 Saga 패턴 보상 트랜잭션

- Step 1: 좌석 상태 SOLD 변경
- Step 2: 결제 요청 → 실패 시 Step 1 보상 (좌석 AVAILABLE 복구)
- Step 3: Booking 레코드 생성 → 실패 시 Step 1·2 보상
- 멱등성 키(UUID)로 중복 결제 요청 방어

## 6. 개발 일정 (10주 플랜)

| 단계 | 기간 | 주요 작업 |
|------|------|-----------|
| 1단계 | 1~2주차 | 환경 설정, ERD 설계, 인증(JWT), 이벤트 CRUD |
| 2단계 | 3~4주차 | 좌석 선점 (Redis 분산락), WebSocket 실시간 좌석 맵 |
| 3단계 | 5~6주차 | 대기열 시스템 (Redis Sorted Set + SSE + 입장 토큰) |
| 4단계 | 7~8주차 | 결제 / Saga 패턴, 예매 취소·환불 로직 |
| 5단계 | 9~10주차 | 성능 테스트 (JMeter), 쿼리 최적화, Docker 배포 |

### MVP 범위 (필수)

- 이벤트 조회 / 좌석 선택 / 예매 / 결제 (Mock) / 마이페이지
- Redis 분산락 기반 동시성 제어
- WebSocket 실시간 좌석 현황
- Redis 대기열 + SSE 순번 알림 + 입장 토큰 검증

### Optional 기능

- 실제 PG 연동 (토스페이먼츠 등)
- JMeter 부하 테스트 결과 문서화
- Kubernetes 배포 (EKS / GKE)
- 모니터링 (Prometheus + Grafana)
